## Deployment Architecture

ProcureAI is designed for cloud-native deployment using managed services that minimize operational overhead:

| Component | Service | Tier | Purpose |
|---|---|---|---|
| Frontend (Next.js) | Vercel | Pro | SSR, static assets, edge caching |
| Backend (FastAPI) | Railway or Fly.io | Pro | API server, LangGraph pipeline execution |
| MCP Servers (TypeScript) | Railway or Fly.io | Pro | 4 MCP servers run as subprocess children of the backend |
| Database (PostgreSQL) | Supabase Cloud | Pro | Persistent storage, Row Level Security, real-time subscriptions |
| LLM Provider | OpenAI API | Pay-per-token | GPT-4o-mini for all agent reasoning |

For on-premise requirements, all components can be self-hosted: Next.js on any Node.js server, FastAPI behind Nginx/Gunicorn, PostgreSQL on bare metal, and Llama 3.1 on local GPU hardware. No cloud dependency is mandatory.

## Monitoring and Observability

ProcureAI includes built-in operational telemetry at every layer:

| Layer | Tool | What It Captures |
|---|---|---|
| Infrastructure | Supabase Dashboard | Database health, connection pool, storage usage, query performance |
| Backend API | FastAPI `/health` endpoint | Service status, MCP server connectivity, database reachability |
| Pipeline Execution | Decision Log table (`decision_log`) | Every agent decision, input/output, timestamps, approval status |
| Error Tracking | Structured logging (Python `logging`) | Exceptions, MCP call failures, LLM timeouts |
| Uptime | External health check (UptimeRobot / Better Uptime) | HTTP availability, response time trends, alerting |

The `decision_log` table serves as both an audit trail and operational telemetry source. Every PO generated by the pipeline records the full chain of agent reasoning, enabling post-hoc analysis of system behavior without additional tooling.

## Integration Points

| System | Integration Method | Direction | Notes |
|---|---|---|---|
| Cummins ERP (SAP) | MCP server (erp-data) | Read | Product catalog, inventory levels, forecast data |
| Supplier Portal | MCP server (supplier-data) | Read | Supplier profiles, pricing, lead times, scores |
| Logistics/TMS | MCP server (logistics) | Read | Container specs, shipping rates, route data |
| PO System | MCP server (po-management) | Read/Write | PO creation, status updates, approval workflow |
| Email/Notifications | FastAPI webhook (future) | Write | Approval requests, PO confirmations |

All integrations use the MCP JSON-RPC protocol over stdio, making it straightforward to swap data sources (e.g., replace mock data with live SAP API calls) without modifying the agent or pipeline code.
